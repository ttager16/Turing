Problem Statement:
## Problem Statement

You are building a critical component for a large-scale distributed database system. The system manages shards (data partitions) across multiple data centers, and you need to track which shards are connected in the same logical group (component) for routing query traffic efficiently.

Your task is to implement a **versioned disjoint-set (union-find) data structure** that supports:

* **Union operations** that merge shard groups, subject to load capacity constraints
* **Split operations** that disconnect specific shards from their groups
* **Find operations** that check connectivity in the current state
* **Versioned find operations** that check historical connectivity at specific points in time

The system must maintain a complete version history, allowing queries to examine the connectivity state as it existed after any previous operation completed.

---

## Objectives

### 1. Implement Core Disjoint-Set Operations with Load Constraints

Implement a union-find data structure where:

* Each shard has a fixed **integer load value** calculated as `(shard_id % 100) + 1` (range: 1 to 100)
* Each component maintains the **sum of all shard loads** within it
* Union operations must enforce a **maximum component load threshold of 500**

**Union Operation Behavior:**

* When merging two components, calculate their combined load
* If `combined_load > 500`, reject the union and return `"False"`
* If `combined_load ≤ 500`, perform the merge and return `"True"`
* Use efficient union strategies (union-by-rank or union-by-size)
* If shards are already in the same component, return `"True"` (no-op success)

---

### 2. Implement Split Operations with Isolation Semantics

Implement a split operation that disconnects a single shard from its component:

**Split Operation Behavior:**

* `split a b` checks if shards `a` and `b` are in the same component
* If they are **not connected**, return `"False"`
* If they **are connected**:

  * Isolate shard `b` into its own singleton component
  * All other shards (including `a`) remain in the original component
  * Return `"True"`
* Update component loads accordingly (original component loses `b`’s load)

---

### 3. Support Version History and Historical Queries

Maintain a complete version history of the data structure state:

**Version Numbering:**

* Version 0: Initial state (all shards are singletons)
* Version k: State immediately after operation (k-1) completes
* Only **union** and **split** operations modify state and increment version
* **find** and **find_versioned** operations are read-only

**Versioned Find Operation:**

* `find_versioned a b v` returns whether shards `a` and `b` were connected at version `v`
* Must query historical state without modifying current state
* Handle queries to any valid past version

---

### 4. Handle Edge Cases and Invalid Operations

**Non-existent Shards:**

* Automatically initialize any referenced shard as a singleton component with its calculated load

**Self-Operations:**

* `union a a` → `"True"` (already connected)
* `split a a` → `"False"` (cannot split from itself)
* `find a a` → `"True"` (always connected to itself)

**Invalid Version Queries:**

* `find_versioned a b v` where `v > current_version` → `"False"`

---

### 5. Optimize for Performance

Achieve efficient time complexity:

* Union and find operations: **amortized O(α(n))** using path compression and union-by-rank
* Split operations: **O(log n)** per operation
* Version queries: Efficient access to historical states
* Memory overhead for version history should be reasonable

---

## Constraints

* The system must handle up to **2,000 shards**
* Operations list may contain up to **10,000 operations**
* Shard IDs are non-negative integers: `0 ≤ shard_id < 2,000`
* Each shard has load: `(shard_id % 100) + 1`, giving values from 1 to 100
* Maximum component load threshold: **500**
* **No external libraries** allowed for data structures; implement all structures manually
* Thread-safety is **not required** (single-threaded execution)
* Must maintain version history without corrupting current or past states
