## Problem Statement

A fast-growing smart-home IoT platform must maintain a **multi-layer, branchable “adaptive linked list”** to ingest high-frequency sensor events (temperature, humidity, light) across multiple building zones (e.g., `zone1`, `zone2`, `kitchen`, `livingroom`). The structure must *look like* a linked list externally (single `next` traversal), but internally allow **hierarchical segmentation** (root → zones → types), **adaptive hot-segment promotion** for bursty sensors, **partial rollback**, and **segment-level queries**—all with **tight memory** via on-demand child creation.

Unlike a traditional linked list, each node may have multiple children (tree-like), but the data structure **exposes a deterministic single linear traversal order** via a `next` pointer computed from a fixed rule so tests can verify correctness.

### Deterministic Rules (Must Implement)

1. **Hierarchy & IDs**

   * Parse each sensor key as `<type>_<zone>` (e.g., `temperature_zone1` → type=`temperature`, zone=`zone1`).
   * Create node IDs as:

     * Root: `"root"`
     * Zone: `"zone:<zone>"`
     * Type under a zone: `"zone:<zone>/type:<type>"`
     * **Hot segment (promotion rule below):** `"zone:<zone>/type:<type>*"` (note the trailing `*`).
2. **On-Demand Children**

   * Create zone and type nodes **only when first needed**.
3. **Hot-Segment Promotion (Adaptive Reorg)**

   * For each `(zone, type)`, when its **event count reaches ≥ 2**, **promote the most recently inserted reading** into a **hot segment** with ID `"zone:<zone>/type:<type>*"`.
   * The base type node **retains all readings**, including those represented in the hot segment; the hot segment **duplicates only the latest reading** for O(1) hot retrieval.
   * Only **one** hot segment per `(zone, type)`; it always mirrors the latest reading.
4. **Partial Rollback**

   * If an event index appears in `invalidations`, **remove that reading** from **both** the base type node’s buffer and the hot segment (if it was the latest when applied). The *latest* valid reading after rollback becomes the hot segment’s value (or the hot segment is deleted if none).
5. **Linked-List Traversal Order (Deterministic)**

   * Linear traversal order (and thus `next` pointers) is:
     `root` → **zones in ascending lexicographic order** → for each zone: **types in ascending lexicographic order** → **its hot segment immediately after its base type node** (if exists).
   * The final node’s `next` is `None`.
6. **Aggregates**

   * For each **type node** (not the hot node), compute:

     * `count`, `sum`, `min`, `max`, `avg` (rounded to **3 decimal places**) over **valid** readings after rollback.
   * For each **hot node**, store only:

     * `latest_value` (float) or `null` if none (and delete the hot node if none).

### Constraints

* No external libraries (no NumPy, no `collections.deque`, etc.).
* On-demand child allocation only; avoid pre-allocating sparse segments.
* Promotion/rollback must keep the structure and outputs **exactly** consistent with the rules above.
* Assume events could arrive concurrently, but the test harness serializes them in the provided order. Your structure should be **thread-safe by design**, but the **deterministic output is judged only on the given sequential order**.
* Use only Python’s basic synchronization primitives if you need them (`threading.Lock`, etc.). No external libs.
