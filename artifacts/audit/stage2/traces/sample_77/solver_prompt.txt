### Problem Statement

In a local computing environment, an ML engineer needs to coordinate a workflow of computational tasks distributed across multiple processing clusters. Each cluster has a limited capacity and may handle multiple dependent tasks at once. The engineer aims to design a scheduling system that can efficiently:

1. Manage dynamic task groups with changing capacities.  
2. Merge or split clusters based on workload conditions.  
3. Adjust task priorities to achieve balanced utilization and minimal delay.

To achieve this, the engineer plans to use a graph-based scheduling structure that models clusters and their relationships, supporting efficient updates and priority adjustments.

### Objectives

1. **Graph-Based Resource Clusters**  
   Each cluster should be represented as a node in a virtual graph. Tasks can be assigned to clusters, and clusters may be linked, merged, or split dynamically to reflect changing workloads.

2. **Efficient Priority Management**  
   A tree-based data structure (for example, a custom segment tree or balanced priority tracker) should be used to efficiently handle capacity updates and priority adjustments. Each update or query should run in O(log n) time or better.

3. **Cluster Operations**  
   The scheduling system must support the following operations:

   - link(a, b): Connect two clusters to form a workload relationship.  
   - split(a): Split cluster "a" internally into two equal subparts. The cluster retains the same external identifier "a", but its capacity is halved (rounded down). The discarded half is no longer tracked.  
   - merge(a, b): Merge clusters "a" and "b" if their combined capacity does not exceed a defined global merge limit of 100. The merged cluster keeps the name of the first operand "a", and cluster "b" is removed from the active cluster set.  
   - adjust(value, a): Adjust the capacity of cluster "a" by "value". A positive value increases it, and a negative value decreases it, if the computed value is less than zero, it must be clamped to zero.
   - balance(a, b): If one cluster’s capacity exceeds 1.5 times that of the other, transfer half the difference in capacity from the larger cluster to the smaller one.

### Constraints

- The implementation must be written in pure Python with no external libraries.  
- Only the following operation types are allowed: "link", "split", "merge", "adjust", and "balance", if any other operation appears in the input, the function must immediately return an empty list [].  
- Payload validation must be strict. Any violation must also return [].  
    1. For link, the payload must be a list [a: str, b: str] where a != b.  
    2. For split, the payload must be a single string "a" that references an existing cluster.  
    3. For merge, the payload must be a list [a: str, b: str] where both clusters exist and a != b, and their combined capacity must not exceed 100.  
    4. For adjust, the payload must be a list [value: int, a: str] where the cluster a exists.  
    5. For balance, the payload must be a list [a: str, b: str] where both clusters exist and a != b.  
- Each operation must be represented as a list of length two [operation, payload].  
- Any operation that references a cluster name not currently present in the active state must return an empty list.
