### Problem Statement

In a metropolitan water-distribution network, a senior infrastructure engineer is responsible for optimizing water flow between multiple reservoirs, purification facilities, and consumption zones. The system is modeled as a directed graph where each node may function as a reservoir, a distribution junction, or a consumption endpoint. Each directed edge represents a pipeline that carries water with a given capacity, which may change due to maintenance, environmental conditions, or operational adjustments. The engineer must compute the maximum feasible flow from a specified source node to a target node after applying all capacity updates. The computation must be deterministic, validation-aware, and fail-fast under any invalid conditions, ensuring consistent and reproducible results for identical inputs.


#### Objectives

1. **Deterministic Flow Computation**
    Implement a deterministic maximum-flow procedure that computes the total feasible flow from source to target, with the following requirements:
      * Traverse adjacency lists in lexicographic string order of node IDs (e.g., "10" < "2").
      * When multiple augmenting paths have equal bottleneck capacity, always select the lexicographically smallest full path.
      * Any internal searches must respect the same lexicographic string order when expanding neighbors.
2. **Batched Capacity Updates**
    * Each update is a list of the form ["u", "v", delta].
    * Edge creation policy:
      * If an edge from u to v does not exist and the update value (delta) is greater than zero, the function must create a new edge with that capacity.
      * If the edge does not exist and the update value is exactly zero, the function must take no action.
      * If the edge does not exist and the update value is negative, the function must immediately return 0.
    * Apply all updates sequentially in order.
    * Negative capacity rule (single authoritative rule): During updates, if any edge's capacity becomes < 0 at any time, return 0 immediately.
    * After all updates, drop (ignore) any edge whose final capacity == 0.
3. **Hard Validation Constraints**
    The algorithm must immediately return 0 without computing flow if any of the following are true:
      * A self-loop exists (e.g., ["X","X",c]).
      * The source or target node is missing or unreachable (no valid path exists) after updates and zero-drops.
      * The graph contains duplicate edges between the same (u,v) pair after dropping zero-capacity edges.
4. **Deterministic State Isolation**
    * All search and residual structures must be freshly initialized for every search/augmentation.
    * No global or persistent residual state may survive between runs.
    * The algorithm must yield identical outputs for identical inputs.
    * No randomization or reliance on unordered dictionary traversal.
