Problem Statement:
In a high-frequency trading environment, a Python developer is responsible for building an **order-flow optimizer** that routes multiple asset trades through a **dynamic network of liquidity providers**.
The goal is to find **minimal-cost trade routes**, **balance multiple currency pairs**, and **track partially filled orders**, all while ensuring **data integrity** during frequent network updates.

The solution must use only **Python’s standard library**, especially the **`copy` module**, to replicate and manage different states of the trading graph safely.
No external packages are allowed.

### **Objectives**

1. **Build a Dynamic Multi-Asset Graph**
   Construct a **directed graph** that models the trading network:

   * Each **node** represents a liquidity provider or exchange.
   * Each **edge** represents a directed trade route between two nodes.
     The edge is defined as a list `[start_node, end_node, capacity, cost]`.

     * `start_node`: integer ID of the source node.
     * `end_node`: integer ID of the target node.
     * `capacity`: maximum trade volume allowed through this route.
     * `cost`: transaction fee per unit of trade volume for this edge.

   The graph should be updatable as trades are added, cancelled, or modified in real-time.

---

2. **Implement a Min-Cost Flow Algorithm with Partial Fill Tracking**
   Develop an algorithm that:

   * Finds the **minimum total cost** required to transfer a given trade volume from its source node to target node.
   * Supports **partial fills**: when only part of the requested volume can be executed due to capacity limits.
   * Processes multiple trades one after another (sequential approximation).
     The flow computed for one trade must update the residual graph before the next trade is processed.

   **Trade Format:** Each trade is represented as a dictionary:

   ```python
   {"volume": <float>, "source": <int>, "target": <int>}
   ```

   **Partial Fill Definition:**
   A partial fill occurs when the total available capacity in the network is **less than the trade’s requested volume**.
   In such cases, the optimizer should allocate as much as possible and record the unfilled portion.

---

3. **Ensure Data Integrity Using the Copy Module**

   * Use Python’s built-in **`copy`** module to duplicate graph states safely.
   * Maintain separate copies for:

     * The **original graph** (baseline).
     * The **intermediate states** (after each trade allocation).
     * The **final state** (after all trades are processed).
   * This ensures that edge reweighting, capacity reductions, or node removals during one trade do not affect others unintentionally.

---

4. **Handle Updates and Edge Conditions Safely**
   The algorithm should:

   * Support addition or removal of edges between computations (representing liquidity changes).
   * Gracefully handle:

     * Self-loops (edges from a node to itself).
     * Zero-capacity edges.
     * Invalid trade definitions (e.g., negative volume).
     * Sudden cost changes (simulate volatility).
   * These should not cause crashes or data corruption.

### **Explanation of Sample Input and Output**

1. **Trade 1:**

   * Requested: 8 units from node 0 → 3.
   * Available path: 0 → 1 → 2 → 3 with total capacity 5 units.
   * Only 5 units can be filled.
   * Remaining 3 units become a **partial fill**.
   * Cost for this path = (2 + 1 + 3) × 5 = 30.

2. **Trade 2:**

   * Requested: 5 units from node 0 → 5.
   * Available paths:

     * Path 0 → 4 → 5 (4 units, cost 2 + 4 = 6 per unit).
     * Path 0 → 1 → 5 (1 unit, cost 2 + 6 = 8 per unit).
   * Total flow = 4 + 1 = 5 units (fully filled).
   * Total cost = (4×6) + (1×8) = 32.

3. **Totals:**

   * Combined flow = 5 (Trade 1) + 5 (Trade 2) = 10 units.
   * Combined cost = 30 + 32 = 62.
   * Flow distribution and partial fills are reported accordingly.
