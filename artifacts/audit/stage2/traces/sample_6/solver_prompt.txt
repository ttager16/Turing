### Time and Capacity Management Details

**Daily Capacity Changes and Time Windows Relationship:**
- The `daily_capacity_changes` list in each commodity dictionary represents capacity modifications for each day of the simulation
- Each element in the list corresponds to a specific day (index 0 = day 0, index 1 = day 1, etc.)
- The `time_windows` parameter provides delivery deadlines for each commodity (by index)
- The algorithm processes commodities in priority order, and for each commodity:
  1. Applies the corresponding time window constraints (affecting edge costs and capacities based on urgency)
  2. Iterates through the `daily_capacity_changes` list, applying capacity degradation for each day
  3. The capacity degradation is cumulative: each day's degradation builds upon previous days' effects

**Time Window Impact on Routing:**
- Perishables with tight time windows (< 2 days) get 50% capacity reduction and 50% cost increase
- Pharmaceuticals with tight time windows (< 3 days) get 20% capacity reduction and 20% cost increase
- These modifications ensure urgent deliveries are prioritized but at higher cost

**Priority vs. Cost Optimization Resolution:**
The algorithm resolves conflicts between priority-based routing and cost minimization using the following hierarchy:
1. **Priority Processing Order**: Commodities are processed in strict priority order (electronics → pharmaceuticals → perishables → textiles → chemicals)
2. **Path Selection Within Priority**: For each commodity, the algorithm finds the minimum-cost path that satisfies all constraints (capacity, cross-contamination, time windows)
3. **Constraint Enforcement**: If a higher-priority commodity blocks a lower-priority commodity's preferred route, the lower-priority commodity must find an alternative (potentially more expensive) route
4. **Cost Minimization**: Within the constraints imposed by priority ordering and cross-contamination rules, the algorithm minimizes total transportation and storage costs
5. **Fallback Strategy**: If a commodity cannot meet its minimum flow threshold (20% of demand) due to constraints, its flow is set to zero rather than compromising higher-priority commodities

**Cross-Contamination Prevention Implementation:**
The cross-contamination prevention is implemented through the following mechanisms:
- **Incompatible Type Mapping**: 
  - Chemicals are incompatible with perishables and pharmaceuticals
  - Perishables are incompatible with chemicals
- **Node Usage Validation**: Before routing a commodity through a node, the algorithm checks:
  1. If any incompatible commodity types are currently using that node (direct usage)
  2. If any incompatible commodity types are using adjacent nodes (within distance 1)
- **Adjacent Node Restriction**: The "separated by at least one intermediate node" requirement is enforced by preventing incompatible commodities from using nodes that are adjacent (within distance 1) to nodes used by incompatible types
- **Real-time Constraint Checking**: During path finding, each potential node is validated against existing commodity flows to ensure no cross-contamination occurs

### Return Value
The function returns a list:
1. A 2D list representing the multi-commodity flow solution; each sub-list corresponds to a commodity's flow distribution across all edges in the same order as they appear in the base_graph adjacency list (flattened by iterating through each node's neighbors sequentially).  
2. A float representing the total minimized cost (including storage and transit).

**Cost Calculation Details:**
- **Transportation Cost**: Sum of (flow_amount × edge_cost) for all edges used
- **Storage Cost**: Additional 0.1 cost per unit flow for intermediate nodes (nodes that are neither source nor sink)
- **Total Cost**: Transportation cost + Storage cost

**Flow Matrix Generation:**
- The flow matrix has dimensions [num_commodities × total_edges]
- Edges are indexed in the order they appear in the base_graph adjacency list (flattened)
- Each element flow_matrix[i][j] represents the flow of commodity i on edge j
- The matrix is generated by iterating through all edges and matching them with commodity flows
