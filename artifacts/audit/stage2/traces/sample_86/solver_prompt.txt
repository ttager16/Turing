Problem Statement:
### Problem Statement

A **decentralized cryptocurrency exchange** operates across multiple blockchain networks. Each network has varying **block confirmation times** and **transaction finality mechanisms**, which can cause updates to arrive out-of-order. The exchange maintains an **order book aggregator** that continuously synchronizes **price and volume data** from hundreds of trading pairs, while also detecting **cross-chain arbitrage opportunities**.

The core data structure is a **segment tree** that tracks:

* **Current prices** of assets
* **Cumulative transaction volumes**
* **Liquidity depth changes**
* **Price volatility** over overlapping time windows

### Challenges

1. **Out-of-Order Updates (Blockchain Reorganizations / Reorgs):**

   * Due to blockchain reorganizations, previously confirmed transactions can be invalidated.
   * When a reorg occurs, all updates in a given timestamp range must be rolled back. Any new or corrected updates should be applied via separate update operations.

2. **Conditional Updates:**

   * Some updates only take effect if certain thresholds are crossed in adjacent segments (e.g., sudden price spikes or low liquidity).
   * This creates **dependencies between non-contiguous nodes**.

3. **Multi-Metric Aggregation:**
   Each node in the segment tree maintains:

   * **Weighted Average Price (WAP)** for a range,  calculated as `sum(price_i * volume_i) / sum(volume_i)`

   * **Maximum Price Deviation (MPD)** for a range, calculated as `max(price_i) - min(price_i)`

   * **Cumulative Volume (CV)** for a range, calculated as `sum(volume_i)`

   * **Liquidity Score (LS)**, calculated as `LS = CV / (1 + MPD)`

   * Range queries must return **all four metrics simultaneously**.

   * Lazy propagation is required, but derived metrics (like LS) depend on multiple underlying metrics, complicating updates.

4. **Snapshot and Restore:**

   * The entire tree state can be checkpointed at any time.
   * Snapshots allow the system to revert to a previous exact state, discarding intermediate updates.
   * Multiple snapshots can coexist, identified by unique timestamps, and must correctly restore all metrics, including partially propagated values.

### Objectives

1. **Implement Multi-Metric Segment Tree with Interdependent Aggregations:**
Design a segment tree where each node maintains WAP, MPD, CV, and LS. Parent nodes must correctly aggregate these metrics from children, accounting for the non-linear relationship in the liquidity score formula.

2. **Support Out-of-Order Updates with Timestamp-Based Rollback:**
Allow updates tagged with timestamps. When a blockchain reorg is detected (indicated by a rollback command with a timestamp range), all updates within that range must be undone in reverse chronological order, then new corrected updates applied.

3. **Handle Conditional Propagation Across Non-Contiguous Segments:**
The `"conditional_check"` operation simulates an adjustment to the specified source `[i, j]` and destination `[k, l]` ranges when it is processed. 
If MPD in the source range exceeds `mpd_threshold` AND CV in the destination range is below `cv_threshold`, a price adjustment (`price_boost`) is applied to all elements in the destination range.

4. **Snapshot and Restore Mechanism:**
Provide functionality to capture the complete state of all metrics across all nodes at a given moment, assign it a snapshot ID, and later restore to that state, invalidating all updates applied after the snapshot timestamp. 

### Constraints

- The segment tree must support point updates, range updates, and range queries on all four metrics simultaneously.
- Updates arrive with associated timestamps (integers). A rollback command specifies a timestamp range `[t1, t2]`; all updates where `t1 ≤ timestamp ≤ t2` must be undone.
- If total cumulative volume (CV) in a queried range is 0.0, define WAP = 0.0 to avoid division-by-zero.
- When calculating Maximum Price Deviation (MPD), only elements with non-zero cumulative volume (CV) are considered. If all elements in the range have zero volume, MPD should be 0.0.
- Conditional updates (`"conditional_check"`) are evaluated immediately when the operation is processed.
- If MPD in the source range exceeds mpd_threshold AND CV in the destination range is below cv_threshold, price_boost is applied to all elements in the destination range.
- Snapshot restoration must preserve the exact node configuration, including any partially propagated lazy values that existed at snapshot time.
- Restoring an unknown snapshot_id leaves the system state unchanged. Creating a snapshot with a duplicate snapshot_id overwrites the previous snapshot.
- All operations must maintain mathematical precision for floating-point calculations; rounding errors that accumulate across operations are considered incorrect.
- No external libraries; only Python standard library constructs (dict, list, basic math operations) are permitted.
