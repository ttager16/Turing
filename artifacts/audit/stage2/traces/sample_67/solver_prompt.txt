Problem Statement:
### Problem Overview

A multinational logistics alliance manages a **continuously evolving transportation network** spanning thousands of distribution hubs.
Each operational change — creating a new route, adjusting capacity, disconnecting an overloaded path, or merging two strategic timelines — must be tracked in a **fully versioned history**.

Because different divisions experiment with alternative configurations, the system must support **branching** from any prior version, **merging** compatible versions, **splitting** routes, and **querying** both connectivity and route attributes (such as minimum and maximum capacities) — all while preserving full immutability of past states.

Your task is to implement a **multi-version, attribute-aware disjoint-set (persistent union–find)** that supports branching timelines, merges under constraints, and attribute aggregation queries across many historical versions.

---

### Objectives

1. **Persistent Multi-Version DSU**

   * Maintain parent/rank information and route capacities across all versions using **structural sharing**.
   * Every update (`union`, `split`, `branch`, `merge_versions`) must produce a **new immutable version** without copying the entire structure.

2. **Branching and Merging Timelines**

   * `branch(from_version)` creates a new independent version that inherits state from `from_version`.
   * `merge_versions(vX, vY)` attempts to unify two versions into a new one if all overlapping routes satisfy the merge constraint (e.g., route capacity ≥ 0).

3. **Attribute Propagation and Queries**

   * Each added route carries an integer `capacity`.
   * `attributes(a, b, version)` must return `[min_capacity, max_capacity]` along the connection path between hubs `a` and `b` in that version.
   * If hubs are disconnected, return `[]`.
   * If `a == b`, return `[0, 0]`.

4. **Splits and Partial Reversions**

   * `split(a, b, version)` removes the **direct** route between `a` and `b`, creating a new version with updated connectivity and attributes.

5. **Deterministic Versioning**

   * Versions are numbered sequentially starting from `0`.
   * Each mutating operation (`union`, `split`, `branch`, `merge_versions`) produces the next version ID deterministically (`max_version + 1`).
   * Queries (`find`, `attributes`) do not create new versions.

---

### Constraints (realistic and aligned with implementation)

* Up to `10^5` hubs and `5×10^5` total operations.
* Each version is a **forest** (no cycles per version). Unions add exactly one tree edge; merges do not create cycles.
* Connectivity and attribute queries operate over a **unique tree path** using persistent binary lifting.
* No path compression (to preserve immutability and LCA tables).
* Approximate time bounds in pure Python:

  * `union` / `find`: **near-logarithmic**, bounded by tree height.
  * `branch`, `split`, `merge_versions`: **near-logarithmic** (structural sharing overlaid maps).
* No full O(N) copies per version. Use structural sharing (copy-on-write overlays).
* Only **built-in Python** is allowed (no external libraries).
* Deterministic version IDs: each mutating operation creates exactly one new version.
