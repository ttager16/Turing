## Problem Statement
An intelligent transportation research team is developing an advanced ride-sharing optimization system for urban mobility. The system models the dynamic availability of drivers and passengers as a temporal bipartite graph where connections between drivers and passengers vary based on time windows, geographical proximity, and service constraints.

You are tasked with implementing a sophisticated Temporal Bipartite Matching algorithm that processes real-time driver-passenger data to maximize successful ride matches while minimizing overall passenger wait times. The system must handle time-dependent availability windows, distance constraints, and capacity limitations across multiple service zones within the urban area.

## Core Requirements
Your solution must implement a **pure Python** Temporal Bipartite Matching algorithm (no external libraries) that processes urban ride-sharing data and generates optimal driver-passenger assignments. The algorithm should prioritize matches based on efficiency metrics while respecting all temporal and geographical constraints.

### Key Operational Constraints
1. **Temporal Availability Constraints**: Both drivers and passengers have specific time windows during which they are available for matching
2. **Geographical Distance Constraints**: Maximum allowable distance between driver and passenger pickup locations
3. **Vehicle Capacity Constraints**: Each driver has a maximum passenger capacity limit
4. **Service Zone Constraints**: Drivers can only serve passengers within designated service zones
5. **Priority Level Constraints**: Premium passengers (lower priority numbers) must be prioritized for matching

### Advanced Algorithmic Requirements
The Temporal Bipartite Matching must incorporate a multi-criteria optimization system that evaluates match quality using this composite formula:

**Match Quality Score = Base Distance Cost + (Time Window Overlap Bonus × Priority Weight) + (Zone Efficiency Factor × Capacity Utilization)**

Where:
- Base Distance Cost: Euclidean distance between driver and passenger locations
- Time Window Overlap Bonus: Calculated as (overlap_duration / max_wait_time) × 100, where overlap_duration = min(availability_end, request_time + max_wait_time) - max(availability_start, request_time)
- Priority Weight: (6 - passenger_priority_level) × 15
- Zone Efficiency Factor: Calculated as (1.0 / zone_congestion_level) × 20
- Capacity Utilization: (current_passenger_count / vehicle_capacity) × 10

The algorithm must also implement surge pricing detection where service zones with passenger_to_driver_ratio > surge_threshold have their match quality scores reduced by 15% to encourage driver migration to high-demand areas. The passenger_to_driver_ratio is calculated as (number_of_passengers_in_zone / number_of_available_drivers_in_zone). For surge calculation, passengers are counted based on their declared service_zone field regardless of pickup location coordinates, and drivers are counted based on having the zone in their service_zones list and having remaining capacity > 0.

**Surge Adjustment Formula**:
- If passenger_to_driver_ratio > surge_threshold: adjusted_score = original_score × 0.85
- Otherwise: adjusted_score = original_score

## Implementation Guidelines
**Temporal Bipartite Graph Construction**: Build a bipartite graph where drivers and passengers are separate vertex sets, with edges representing valid matches based on temporal and spatial constraints
**Maximum Weight Bipartite Matching**: Implement the Hungarian algorithm or Ford-Fulkerson method to find a maximum weight matching in the bipartite graph
**Time Window Overlap Calculation**: Calculate overlap between driver availability and passenger request time plus max_wait_time
**Distance Calculation**: Use Euclidean distance formula: sqrt((x2-x1)² + (y2-y1)²) for all location-based calculations
**Zone Boundary Validation**: Ensure pickup locations fall within specified zone boundaries before creating graph edges
**Capacity Constraint Handling**: Only create edges between drivers and passengers if the driver has sufficient remaining capacity
**Surge Detection**: Calculate passenger-to-driver ratio for each zone and apply surge adjustments to match quality scores
**Priority Processing**: Weight match quality scores based on passenger priority levels to favor premium customers
**Maintenance Window Filtering**: Exclude zones that are under maintenance during the current system time
**Deterministic Matching**: When multiple matches have identical quality scores, prioritize by driver_id in ascending order

4. **Surge Adjustment**: Apply surge pricing adjustments to zones exceeding the passenger-to-driver threshold

6. **Distance and Time Estimation**: Calculate pickup distances and estimated travel times for all successful matches
7. **System Metrics Compilation**: Compute efficiency metrics, including utilization rates and average performance indicators
8. **Result Assembly**: Generate a comprehensive output with matches, metrics, and unmatched entities

The algorithm should terminate successfully when all feasible matches are identified within the given constraints, optimizing for both match quantity and quality while respecting all temporal, spatial, and capacity limitations.
