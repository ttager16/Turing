# Problem Statement

In a next-generation urban planning project, geospatial analysts must reconcile overlapping city zones (parks, residential areas, commercial districts) and their resource networks (water mains, major roads, fiber corridors). Full geometric boolean operations and continuous resource-flow simulation are out of scope for this task — instead, you will implement a **deterministic, testable subsystem** that models the *structural* aspects of zone merging using simplified but realistic approximations.

Each input zone is given as an axis-aligned polygon approximation (a list of `(x,y)` vertices forming a rectilinear-like polygon). Resources are represented as simple line segments or numeric aggregate metrics (capacity, density). Your job is to identify which zones should be merged based on *geometric adjacency/overlap* (approximated by bounding-box overlap) and *resource compatibility* (e.g., combined capacity limits), then produce a merged list of zones that conservatively aggregates geometry and resources.

This simplified subsystem preserves the spirit of a larger composite structure (Disjoint-Set for connectivity, plus auxiliary aggregators for resources) while being deterministic, verifiable, and suitable for automated grading.

# Objectives

1. **Connectivity via Disjoint-Set:**

   * Build a union-find (disjoint-set) to group zones that should be merged. Two zones are merge candidates if **their axis-aligned bounding boxes intersect** (non-empty intersection) *and* their resource aggregates satisfy compatibility checks described below.

2. **Resource Compatibility Checks:**

   * For a resource type (e.g., `water`), define a simple compatibility rule: the merged capacity must not exceed a global threshold `MAX_WATER_CAPACITY` (configurable internally). If the capacity would exceed the threshold, the zones should **not** be merged for that resource dimension.
   * For `roads`, merge by summing `density` and concatenating `connections`; no upper bound is enforced unless explicitly provided in the input as `max_density` (optional).

3. **Multi-Phase Merge:**

   * Phase 1: compute bounding boxes and identify raw geometric adjacency pairs.
   * Phase 2: for each adjacency pair, evaluate resource compatibility. If compatible, union their DSU sets.
   * Phase 3: after all unions, aggregate per-set geometry and resources deterministically (see Aggregation rules).

4. **Deterministic Aggregation Rules:**

   * Merged `id`: deterministic new id equal to `min(all constituent ids) * 100 + count` (so it's reproducible).
   * Merged `type`: if all constituent types are identical, keep that type; otherwise set to `'mixed'`.
   * Merged `coordinates`: produce a deterministic concatenation of all input coordinates in ascending order of zone id (no geometric union — just concatenation, which is deterministic and represents the combined shape conservatively).
   * Merged `resources`: for each resource type, apply deterministic aggregation (sum capacities, sum densities, concatenate route lists preserving order).

5. **Extensibility Notes:**

   * Document in comments where a spatial index (R-tree) or an actual polygon union routine would be integrated for higher fidelity.

# Aggregation Rules (Detailed)

* `id`: `new_id = min(ids) * 100 + count` where `count` is the number of merged zones in that set.
* `type`: if all `type` strings equal -> keep; else `'mixed'`.
* `coordinates`: concatenation of coordinates lists in ascending order by original `id` (no deduplication required).
* `resources`:

  * `water`: `capacity` = sum capacities; `routes` = concatenation of route lists preserving zone-id order. If any zone has `max_water_capacity` and the sum exceeds it, those zones should not have been unioned — ensure compatibility check prevents this.
  * `roads`: `density` = sum densities; `connections` = concatenation of all connections.
  * For unknown resource keys, default to conservative aggregation: numeric -> sum, lists -> concatenate, others -> keep as list of values.
