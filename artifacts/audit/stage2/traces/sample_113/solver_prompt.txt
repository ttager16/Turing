## Problem Statement

You are managing a smart city's traffic network during peak hours where multiple vehicle types (commodities) must travel through a directed road network simultaneously. Each road has a total capacity that must be shared among all vehicle types, and certain roads have restrictions on which vehicle types can use them.

## Problem Description

Your task is to compute the maximum total flow across all commodities from their respective sources to a common sink, while respecting individual road capacities, commodity-specific restrictions, and ensuring that flow conservation holds at all intermediate nodes. Additionally, you must identify the minimum bottleneck cut, compute residual graph properties, and provide detailed flow decomposition for traffic management insights.

Given a directed graph with nodes, edges (with capacities and time-varying multipliers), multiple commodity sources, a single sink, road restrictions per commodity, and a flow computation strategy, compute:
1. Maximum total flow achievable across all commodities
2. Minimum cut edges that limit total flow in the residual graph
3. Flow assignment for each commodity on each edge
4. Bottleneck edges operating at full capacity
5. Residual capacity distribution across the network
6. Flow paths decomposed by commodity
7. Critical intersection nodes that affect overall flow
8. Per-commodity flow values from each source to the sink

The solution must handle three algorithmic strategies for flow computation:
1. Edmonds-Karp: Sequential commodity routing using BFS-based augmenting paths
2. Capacity Scaling: Sequential commodity routing with exponential capacity scaling (delta scaling), commodities processed in input order
3. Dinic's Algorithm: Sequential commodity routing using level graphs and blocking flows

All strategies route commodities sequentially in the order provided in the sources and commodities lists (commodity 0 first, then commodity 1, etc.). Each commodity uses the residual capacity remaining after all previous commodities have been routed.

## Output Format

Return a dictionary containing:

- **commodity_flows**: List of integers representing flow value for each commodity (same order as commodities input, each ≤ its demand)
- **min_cut_capacity**: Integer capacity of the minimum cut in the residual graph after routing flows
- **min_cut_edges**: List of [from, to] integer pairs representing edges in the minimum cut
- **source_partition**: Sorted list of node integers reachable from any source in the residual graph after flow routing
- **sink_partition**: Sorted list of node integers in the sink side of the partition
- **flow_assignments**: List of dictionaries, each containing "edge" (list of 2 integers), "flows" (list of integers showing flow per commodity), "total" (integer sum)

## Important Clarifications

**Demand-Limited vs. Capacity-Limited vs. Mixed-Limitation Flows:**

The min-cut is computed in the **residual graph after routing all commodity flows**. The relationship between min-cut and total flow depends on network topology and commodity-specific constraints:

### 1. Pure Demand-Limited Flow

- **All commodities achieve their full demands**
- `min_cut_capacity`: **0** (no bottleneck remains)
- `source_partition`: **all nodes including the sink**
- `sink_partition`: **empty**
- The network could handle more flow if demands were higher
- **Example:** Two commodities each want 10 and the network can handle 100; both get what they requested

### 2. Pure Capacity-Limited Flow

- **At least one commodity cannot meet its demand** and the residual graph has no alternative paths to the sink
- `min_cut_capacity`: **equals `total_flow`** (max-flow min-cut theorem)
- `source_partition` and `sink_partition`: valid s-t cut (sources and sink separated, sink is not reachable from sources in residual graph)
- **Example:** Two commodities demand  but only total 25 can flow due to a cut edge of capacity 25; the cut separates sources from sink

### 3. Mixed-Limitation/Multi-Commodity Topologies

- **At least one commodity is limited (cannot achieve demand) due to a specific bottleneck on its paths**
- However, **the overall residual graph may still have alternative paths to the sink for other flows**
- `min_cut_capacity`: **may be 0** even though not all demands were met
- `source_partition`: **all nodes including the sink**
- `sink_partition`: **empty**
- This is common in multi-commodity settings where *individual commodity paths are restricted but the global network is not fully saturated*
- **Example:** Commodity 1 is limited by a bottleneck edge on its path, but the sink is still reachable through other (possibly unused) paths

> **The min-cut reflects the remaining bottleneck in the residual graph across all flows—not just the bottleneck faced by a single commodity. If the sink remains reachable after all flows, `min_cut_capacity` is 0, even if not all commodities met their demand.**
