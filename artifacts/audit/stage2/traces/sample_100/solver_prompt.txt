## Problem Statement 
MetroTransit’s city network is modeled as a weighted, undirected graph. Nodes are intersections, edges are bi-directional routes with non-negative travel time (edge weight). The network may include disconnected components, isolated nodes, self-loops, and duplicate parallel edges with different weights. Find the global graph center: the node whose maximum finite shortest-path distance (eccentricity) to nodes within its connected component is minimal among all eligible components. To avoid trivial choices, singleton components (isolated nodes) are excluded from the global comparison unless every node in the graph is isolated. If multiple nodes tie, return the smallest node ID.


## Objectives

1. **Normalization DFS-driven**
   * Deduplicate parallel edges by keeping only the minimum weight for each unordered pair {u,v}.
   * Node IDs are strings that represent integers (not contiguous). Treat them as integers for comparisons and tie-breaking.
   * Implement iterative DFS (stack, not recursion) to:
     * enumerate nodes,
     * label connected components,
     * produce per-component node lists.
2. **Candidate Sampling DFS-assisted**
   * For each non-singleton component:
     * Use a DFS-based farthest-point heuristic to pick seed candidates:
       1. run an iterative DFS from any node s (treating edges as unweighted) to find an approximate farthest node a
       2. run DFS from a to get approximate farthest b,
       3. include {a, b} as candidate centers for that component.
     * Also include the component’s highest-degree node and smallest-ID node as candidates.
3. **Distance Computation (weighted)**
   * Use Dijkstra’s algorithm (non-negative weights) to compute shortest paths only from the sampled candidates and any node that survives pruning (below).
   * Maintain distances in integers, choose a large INF (e.g., 10**30) and avoid floating-point.
4. **Center Evaluation & Pruning**
   * The eccentricity of node v is the maximum finite distance to nodes in its component.
   * Use a landmark-bounds style pruning:
     * After a few candidate Dijkstras, keep for each node v:
       * a lower bound LB(v) and an upper bound UB(v) on eccentricity derived from already-run sources,
       * Skip evaluating any v with LB(v) >= current_best_ecc.
     * You must not run Dijkstra from every node, solutions that do so violate the performance requirement.
   * Per component, keep the best candidate(s), globally, compare component winners excluding singleton components.
     If all components are singletons, return the smallest node ID overall.
