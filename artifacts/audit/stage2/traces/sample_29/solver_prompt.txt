Problem Statement:
### Problem Statement
A mission-critical healthcare system monitors patient vital signs across multiple hospital wards using interconnected sensors and medical devices. Your task is to design a fault-tolerant patient monitoring system that maintains data integrity and continuous operation even when devices fail or produce anomalous readings. The system must aggregate sensor data, detect and handle device failures, maintain redundancy, and provide validated patient metrics that healthcare professionals can trust for critical decision-making.

Each ward contains multiple devices monitoring different patients, and devices can fail independently. The system must identify failing devices, route data through backup devices when available, validate readings against expected ranges, and flag anomalies. When a device fails, the system should attempt to compensate using related devices while maintaining a complete audit trail of all failures and recovery actions.

---

#### Objectives:
1. Device Failure Detection: Implement a mechanism to identify failing or malfunctioning devices based on missing data, out-of-range values, or degraded/offline device status.
2. Redundancy Management: Maintain and utilize backup devices for each monitored metric. When a primary device fails, automatically switch to backup devices while tracking which devices are active, failed, or in backup mode.
3. Data Validation and Aggregation: Validate incoming sensor readings against acceptable ranges for each metric type. Aggregate data at ward level by averaging metrics from working devices while handling missing or invalid readings.
4. Fault Recovery Strategy: Activate backup devices when primary devices fail. Implement a strategy to determine when a ward has insufficient working devices (below ward_requirements threshold) to provide reliable monitoring.
5. Audit Trail Maintenance: Track all device failures, backup activations, data validation failures, and recovery actions with timestamps to provide complete traceability for clinical review.

---

#### Constraints:
- Input contains data from up to 50 devices across up to 10 wards
- Each device monitors 1-3 vital sign metrics (heart_rate, blood_pressure, temperature, oxygen_saturation, respiratory_rate)
- Devices can be marked as primary or backup for redundancy
- A metric is considered valid only if at least one device (primary or backup) provides readings within acceptable range
- The system must handle: missing readings, out-of-range values
- Must return detailed failure report if any ward lacks minimum required working devices (specified in ward_requirements)
- Must work with Python standard library only (no external packages)
- Solution should efficiently handle device status changes and reading validation in O(n times m) time where n is devices and m is metrics per device
- Return empty critical_wards list if all wards meet minimum device requirements
- Return None for ward_data of a ward if it has no working devices
- For empty device_readings input, return empty structures for all fields except timestamp
- If a device appears in readings but not in device_config, treat it as a primary device with default behavior
- If a ward is not specified in ward_requirements, use a default minimum of 1 working device

#### Expected Output Format
The output should be a comprehensive dictionary structured as follows:

1. ward_data: For each ward, provide aggregated validated metrics (averaged from working devices), count of working devices, list of active device IDs, backup devices activated if any, and data quality indicator based on the following criteria:
   - "high": working_device_count > ward_requirements (surplus capacity)
   - "medium": working_device_count == ward_requirements (meets minimum exactly)
   - "low": working_device_count < ward_requirements (below minimum, critical)

2. device_status: For each device, indicate current operational status (working/failed), whether it is primary or backup, if backup was activated due to primary failure, and last successful reading timestamp (None if device has failed).

3. validation_failures: List of readings that were rejected including device ID, metric name and value, reason for rejection (out_of_range or missing_data), and timestamp.

4. critical_wards: List ward IDs where working device count fell below minimum requirement specified in ward_requirements, requiring immediate attention. Return empty list if all wards meet requirements.

5. fault_recovery_log: Chronological list of events including:
   - primary_device_failure: when a primary device fails with reasons
   - backup_activation: when a backup device is activated for a failed primary
   - critical_ward_detected: when a ward falls below minimum device requirements

6. timestamp: Unix timestamp when this validation snapshot was completed.

This structure ensures healthcare professionals can quickly assess system health, identify failing equipment, understand data reliability per ward, and review the complete chain of fault detection and recovery actions for clinical audit purposes.
