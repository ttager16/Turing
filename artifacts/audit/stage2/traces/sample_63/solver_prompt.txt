Develop a comprehensive Python-based road network management system for urban infrastructure planning. This system serves as a critical component for city development platforms that require efficient connectivity tracking, capacity management, and dynamic network optimization. The framework must implement a Disjoint-Set (Union-Find) data structure to manage road connectivity through geometric union operations, enabling real-time updates as new roads are constructed, existing roads are modified, and network optimization decisions are made based on priority and proximity metrics.

## Constraints

### General Requirements
1. **Standard Library Only**: Use only Python standard library components.

### Union-Find Implementation
2. **Data Structure**: Maintain parent pointers, rank values, per-node capacities, component sizes, merge history log, and maximum rank tracker (called max_tree_depth for historical reasons, but stores rank values).
3. **Union by Rank**: Attach shorter tree to taller tree root. On equal ranks, make second root the child and increment surviving root's rank by 1. After union, update max_tree_depth to the greater of the two root ranks (note: this tracks maximum rank, which is an upper bound on actual tree depth).
4. **Merge Propagation**: Accumulate capacity and size from child root to parent root. Log merges before structural changes with both nodes. Return boolean indicating if union occurred.
5. **Singleton Creation**: Initialize new nodes with self-parent, rank 0, specified capacity (default 1), and size 1.

### Road Network Management
6. **Road Normalization**: Store roads as sorted tuples (smaller node first) for canonical representation.
7. **Road Priority**: Earlier roads get higher priority. Maintain in a priority queue for efficient retrieval.
8. **Road Insertion**: Initialize both endpoints with make_set using road capacity. Union the nodes. Store road metadata (capacity, active flag, priority). Add to priority queue, adjacency maps, node set, and update total capacity counter.
9. **Intersection Creation**: Use default capacity 1. Add to node set only—no road metadata, priority queue, or adjacency updates. Note: Intersections cannot be targeted by reclassify, expand, or close_partial operations since they lack road metadata.
10. **Dual Capacity Tracking**: Maintain separate capacity systems—union-find tracks component capacities (accumulated during unions), while road map tracks individual road capacities for computing total network capacity.

### Update Operations
11. **Reclassify**: If the specified road exists in road_map, update its capacity. Adjust total_capacity counter by the difference. If road doesn't exist (e.g., it's an intersection), silently skip the operation.
12. **Expand**: Add extended road normally. If original and extended roads share nodes, they're automatically connected. Otherwise, union all non-connected node pairs between the two roads.
13. **Get Neighbors**: Collect all endpoint nodes from roads incident to the given node, excluding the node itself.
14. **Close Partial**: If the specified road exists in road_map, reduce its capacity by specified lanes. If capacity reaches 0, mark inactive and increment deactivated roads counter. If road doesn't exist, skipt it.

### Optimization and Analysis
15. **Priority Roads**: Extract k highest-priority roads from the priority queue. Filter for existence and active status.
16. **Optimize Operation**: Identify priority roads based on k parameter. If boost enabled, increment each priority road's capacity by 1 via reclassify.

### Output Format
17. **Statistics**: Include merge count (merge history length), component count, component sizes, total capacity (sum of active road capacities from road_map), max_tree_depth (maximum rank value), fragmentation index, deactivation count, and component capacities (from union-find, aligned with sorted components).
18. **Return Structure**: Dictionary with 'components' (list of sorted lists) and 'statistics' with specified metrics.
