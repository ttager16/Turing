Problem Statement:
### Problem Statement

A cybersecurity firm is responsible for detecting and preventing multi-layer infiltration attempts across a large communication network.
The network is represented as a **multi-layer directed graph**, where each **node** represents a device, and each **edge** represents a possible communication path between devices.

Unlike a simple network, here each edge belongs to one or more **layers**, representing different infiltration channels (for example: email channel, wireless channel, or internal API access).
These layers can **overlap**, meaning a single device or edge can exist in multiple layers simultaneously.

The firm’s objective is to identify a **minimal set of devices** that must be monitored to block all possible infiltration routes across one or more layers.
This set corresponds to the **Minimal Vertex Cover** — a smallest group of nodes such that every active infiltration edge (communication link) has at least one endpoint in the cover.

The system receives **offline queries**, each query performing one or more of the following operations:

1. Merging multiple infiltration layers to simulate combined attack patterns.
2. Adding or removing edges or nodes in some layers to reflect network updates.
3. Requesting the updated minimal vertex cover for the resulting configuration.

The task is to design an efficient algorithm to compute this adaptive minimal vertex cover for each query while incorporating all updates and merged layers.

---

### Objectives

1. **Compute Minimal Vertex Cover:**
   For each query, compute the minimal vertex cover that covers all active infiltration edges after considering the merged or modified layers.

2. **Handle Layered Updates:**
   Process queries that may introduce new edges, remove old ones, or merge layers to simulate hybrid infiltration paths.

3. **Offline Processing:**
   Handle all queries in an offline manner (i.e., process after collecting all updates).

---

### Constraints

1. Each **layer** is represented independently but can share nodes and edges with other layers.
2. Each **query** includes:

   * A list of **nodes** to be considered in the combined infiltration path.
   * A list of **edges** representing additions, removals, or updates for that query.
3. The function must not use any external Python libraries.
4. The solution must handle:
   * Empty layers
   * Merged layers containing repeated nodes or edges
   * Disconnected subgraphs
5. The output for each query must be the minimal vertex cover represented as a list of node identifiers.

### Computational Complexity and Acceptable Solutions

* **NP-hard acknowledgment:**
  The Minimal Vertex Cover problem is NP-hard. Hence, two solution modes are acceptable:

  * **Exact mode:** Allowed only for small graphs (e.g., ≤ 40 nodes) using exponential algorithms such as backtracking or bitmask DP.
  * **Approximation mode:** For larger graphs, use a 2-approximation (e.g., via maximal matching) and clearly state that the result is approximate.

### Key Definitions

* **Directed Graph:**
  A graph where each edge has a direction, represented as `[u, v]` meaning there is a communication path from node `u` to node `v`.

* **Layer:**
  A distinct logical subset of the network representing a unique infiltration channel (e.g., email, IoT, internal API).
  Each layer may overlap with others — the same node or edge can exist in multiple layers.

* **Minimal Vertex Cover:**
  The smallest set of nodes such that every edge in the current network configuration has at least one of its endpoints included in the set.
  In formula form:
  For all edges `[u, v]`, at least one of `u` or `v` must belong to the vertex cover set.

* **Offline Query:**
  A query that modifies or merges layers but is processed later in sequence rather than immediately after input.

### Input Format

* **graph**:
  A dictionary representing the multi-layer directed graph.

  * Each key is a list `[node, layer_id]` representing a device in a specific layer.
  * Each value is a list of destination nodes (within the same layer).

* **queries**:
  A list of queries.
  Each query is represented as a list of two elements:

  1. A list of nodes to consider in the merged infiltration subset.
  2. A list of edges represented as lists `[u, v]`, describing updates or modifications to the current infiltration graph.

* Layer Merge and Update Rules:
* The nodes and edges provided inside each query are treated as layer-agnostic updates, meaning they apply to all layers that share the referenced nodes.
* When multiple layers contain the same node, any query edge [u, v] merges their connections logically, forming a combined infiltration layer for that query.
* When layers are merged, all directed edges from the selected layers are unified into a single combined graph. The direction of each edge [u, v] is preserved exactly as defined — merging does not imply bidirectional linking unless explicitly specified in the query.
* If a node or edge from the query does not exist in any layer, it should be temporarily added to the merged graph for that query’s evaluation only.
* If an edge [u, v] appears in multiple layers, it is included only once in the merged structure. Temporary nodes or edges introduced by a query exist only for that query’s computation and do not persist into subsequent queries.
* After merging, the minimal vertex cover is computed over this combined graph (without distinguishing between individual layer IDs).

For the purpose of computing the minimal vertex cover, directed edges are treated as undirected connections — meaning that covering either endpoint u or v is sufficient to block the communication path.

#### Formal construction of the combined graph (per query):
1. Start with the union of all directed edges present in any layer that contains at least one node referenced by the query. For each layer entry recorded as "[node, layer_id]": [v1, v2, ...], interpret it as directed edges node -> vi.
2. Apply all query updates (each update is a directed edge [u, v]) by adding those directed edges to the union. The direction of every edge is preserved exactly as given.
3. If an added edge references a node that was not present in any layer, treat that node as temporarily added for this query only. Temporary nodes/edges are not persisted across queries.
4. Remove duplicate edges so each directed pair (u, v) appears at most once in the combined graph. Duplicates across layers do not create multiple parallel edges.
5. For the purpose of computing a vertex cover (see definition), treat every directed edge u -> v as an undirected connection {u, v} — i.e., covering either endpoint blocks that communication path. This does not change the stored combined graph, it only defines how vertex cover is checked.
