### Problem Statement

In a multi-regional securities environment, your task is to design a meldable priority queue in Python that merges and processes concurrent buy and sell order streams from multiple exchanges. Each exchange emits a sequence of unit orders with varying priority levels. The system must seamlessly meld these streams, apply price-based matching where possible, and output a globally ordered list of remaining orders once all eligible trades are executed.


### Objectives

1. **Meldable Structure (from scratch):**
   Implement a Leftist Heap or Pairing Heap with the operations insert, meld, peek_min, and pop_min.
   Use a consistent comparator per heap (one for the buy heap, one for the sell heap) and a single final comparator for global ordering so that merged heaps preserve consistent internal ordering.
   Each operation should maintain approximately logarithmic complexity:
   * insert, pop_min: O(log n) expected
   * meld: O(1) amortized (for Pairing Heap) or O(log n) (for Leftist Heap)
2. **Deterministic Global Ordering:**
   Define the global output ordering key as
   "key = (priority ASC, side ('buy'<'sell'), price_aggressiveness, arrival_index ASC)"
   where:
   * Buys are more aggressive at higher prices (−price in key).
   * Sells are more aggressive at lower prices (+price in key).
   * A monotonically increasing arrival_index (assigned as orders are ingested across all streams) ensures deterministic tie-breaking.
   * Side tie-break rule: 'buy' < 'sell' (i.e., when priority is equal, buys come before sells before considering price).
   * Arrival index assignment: Orders are assigned a strictly increasing arrival_index by iterating through order_streams in order (outer loop over streams, inner loop over orders within each stream). This produces a deterministic, stream-major ingestion sequence.

3. **Matching Rule (unit-volume):**
   Each order is a list [priority, side, price] representing a unit-volume order.
   Maintain two global meldable heaps:
   * A buy heap (max-heap by price; tiebreak by priority ASC then arrival_index ASC).
   * A sell heap (min-heap by price; tiebreak by priority ASC then arrival_index ASC).
   Execute a trade whenever best_buy.price >= best_sell.price, remove one order from each side per match.
   When no further crossing is possible (max(buy.price) < min(sell.price)), stop matching.
   The remaining orders from both heaps are combined and returned in global priority order using the same key.
   *Implementation note:* the buy heap may be represented as a min-heap by storing (-price, priority, arrival_index) so that peek_min and pop_min remain consistent with the required interface.

4. **Meld Flow (single-writer):**
   Each exchange stream is first ingested into its own heap (buy and sell heaps separately).
   All per-stream buy heaps are then melded into one global buy heap, and all per-stream sell heaps are melded into one global sell heap.
   Matching and updates occur in a single-writer phase (conceptual only, no multithreading required) to maintain atomic updates during merges and pops.
