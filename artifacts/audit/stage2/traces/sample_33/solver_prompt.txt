### Problem Statement
In a smart city environment, urban developers aim to manage and optimize traffic flow across a network of roads and intersections. The goal is to create a Python-based algorithm that processes live sensor data from multiple intersections, dynamically adjusts traffic signal timings, and suggests alternate routes to reduce congestion and improve travel efficiency. The algorithm should analyze vehicle density, speed, and congestion metrics to make real-time, data-driven decisions for smoother city-wide mobility.

---

### Objectives

1. Build a simple traffic model that uses sensor data to monitor vehicle counts and congestion levels at each intersection.
2. Automatically adjust signal timings based on congestion and flow rate using a proportional control mechanism.
3. Ensure that the system updates within one second in response to new data while keeping computation efficient.

---

### Constraints

* Input includes sensor data from multiple intersections (each with `vehicle_count`, `average_speed`, and `congestion_level`).
* The algorithm must handle up to 100 intersections efficiently.
* Processing time per update must be under one second.
* Alternate route suggestions should reflect realistic, less-congested paths.

---

### Proportional Control Definition

The signal adjustment factor should be calculated using the following formula:

```python
signal_adjustment = max(0.1, min(1.0,
    1 - (0.5 * congestion_level + 0.3 * (vehicle_count / 300) - 0.2 * (average_speed / 60))
))
```

This ensures higher signal durations (closer to 1.0) for intersections with higher congestion and vehicle counts but lower average speeds.

---

### Route Generation Definition

The city’s intersections are modeled as a **simple sequential network**:

```
A1 → B2 → C3 → D4 → E5 → F6 → ...
```

**Alternate Route Rules:**

* Identify the **most congested intersection** in the dataset.
* For each other intersection, suggest an alternate route that bypasses the most congested one.
* The alternate route should connect to the next **two consecutive intersections ahead** in the sequence, if available.
* If there are **fewer than three intersections**, or if the next two nodes are not available, no alternate route should be generated for that intersection.

**Example Logic:**

* If “B2” is the most congested intersection, then routes from other intersections should skip “B2” and connect through the next two available nodes.
* For instance, if the sequence is A1 → B2 → C3 → D4 → E5, and B2 is most congested:

  * Alternate route for A1 = “From A1 to D4 via C3”
  * Alternate route for C3 = “From C3 to E5 via D4”
  * Intersections near the end (with not enough following nodes) will not have an alternate route.
