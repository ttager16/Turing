### Problem Statement

A smart city infrastructure manages energy consumption across multiple zones with sophisticated optimization requirements. Your task is to design an energy optimization system that uses bitmask operations combined with constraint satisfaction and backtracking to efficiently manage device states across zones. The system must handle device dependencies, priority constraints, shared resource limitations, inter-zone power transfer constraints, weather-based adjustments, state transition costs, mutual exclusion groups, cascading state propagation, load balancing requirements, and device synchronization constraints while minimizing both energy consumption and weighted state transition costs.

Each zone contains devices that can operate in four power states: off (00), standby (01), active (10), and peak (11), encoded using 2 bits. The challenge is that zones share power grid segments with both capacity limits and load balancing requirements, creating coupling constraints where total consumption across specific zone groups cannot exceed segment capacities and must maintain balanced distribution. Additionally, devices have transition cost matrices where moving from one state to another incurs different costs, mutual exclusion groups where at most one device can be in peak state, cascading propagation rules where state changes trigger requirements in dependent devices, and synchronization groups that must change states together.

---

#### Objectives:

1. Bitmask State Management: Implement efficient encoding and decoding of device states where each device occupies 2 bits in a zone's bitmask. Support up to 16 devices per zone (32-bit integer). Use bitwise operations (AND, OR, XOR, shifts) for state extraction, modification, and bulk operations across device groups.

2. Multi-Constraint Satisfaction: Handle eight constraint types simultaneously: (a) device dependencies within zones, (b) priority devices with minimum states, (c) zone energy quotas, (d) shared grid segment capacity limits with load balancing across zone groups, (e) inter-zone service dependencies, (f) mutual exclusion groups where at most one device per group can be in peak state, (g) cascading state propagation where state increases trigger requirements in dependent chains, (h) synchronization groups where devices must transition states together.

3. Optimization with Trade-offs: Minimize weighted objective function combining total energy consumption (weight 0.7) and total state transition costs (weight 0.3). State transition costs vary by device and transition type (e.g., off to peak costs more than standby to active). When multiple solutions have equal weighted cost, prefer solutions with better load balancing across grid segments measured by standard deviation of segment utilization.

4. Backtracking Search with Pruning: Implement search algorithm (backtracking with intelligent pruning, branch-and-bound, or constraint propagation with conflict-directed backjumping) to find valid configurations when greedy approaches fail. Handle cases where satisfying one constraint temporarily violates another, requiring systematic exploration with early pruning of infeasible branches using constraint propagation.

5. Inter-Zone Coordination and Cascading Effects: Process zones considering coupling through shared grid segments and cascading state propagation. When device state increases, all devices in its propagation chain must satisfy minimum state requirements. Handle synchronization groups that must transition together, ensuring atomic state changes across multiple devices potentially in different zones.

6. Load Balancing and Mutual Exclusion: Ensure grid segments are not only within capacity but also balanced (no segment should exceed 80 percent of capacity while another is below 40 percent unless necessary). Handle mutual exclusion groups where peak state assignment requires resolving conflicts and selecting optimal device for peak operation based on energy impact and dependency satisfaction.

#### Constraints:

- System must handle up to 50 zones with up to 16 devices each
- Each device state encoded in 2 bits: 00 (off), 01 (standby), 10 (active), 11 (peak)
- Zone bitmask is 32-bit integer encoding all 16 device states
- Energy consumption per state: off equals 0, standby equals 1, active equals 3, peak equals 5 units
- State transition costs: Transition cost matrix per device where cost(state_from, state_to) varies. Default costs: same state equals 0, adjacent states equals 1, skip one state equals 3, off to peak or peak to off equals 5
- Device dependencies: if required_device has state greater than 0, dependent device must be at least min_state. If required device is 0, dependency does not apply
- Cascading propagation: if device A increases state and device B is in its propagation chain, B must satisfy minimum state equal to A's new state minus 1 (minimum 1)
- Priority devices have minimum required states that cannot be violated under any circumstances
- Grid segments: groups of zones share power capacity with combined energy limit and load balancing requirement (max utilization minus min utilization should not exceed 40 percent unless unavoidable)
- Inter-zone dependencies: [zone_a, device_a, zone_b, min_state_b] means if device_a in zone_a has state greater than 0, the corresponding device_a in zone_b must be at least min_state_b
- Mutual exclusion groups: within each group (can span multiple zones), at most one device can be in peak state (11). If multiple devices need peak, select one that minimizes total energy impact
- Synchronization groups: devices in same sync group must transition to same state simultaneously. All devices in sync group must end at same state
- Weather conditions force certain devices to minimum active state (2) when temperature exceeds threshold
- Must work with Python standard library only
- Algorithm must use backtracking or systematic search with constraint propagation when greedy fails
- Time complexity: O(n times d times 2 to the power k times m) where n is zones, d is devices per zone, k is backtrack depth bounded by 8, m is average constraint checks per state
- Return empty list [] if no valid configuration exists satisfying all constraints
- Return -1 for specific zone if that zone has internally contradictory constraints (e.g., priority devices require more energy than quota, or sync group has conflicting priority requirements)
- Weighted objective function: 0.7 times total_energy plus 0.3 times total_transition_cost
- When multiple valid solutions exist with same weighted cost (within 0.01 tolerance), prefer solution with better load balance (lower standard deviation of grid segment utilization percentages)
- Grid segment violations should be resolved by reducing non-priority devices across affected zones, respecting mutual exclusion and synchronization constraints
- Cascading propagation must be resolved iteratively until fixed point where no new requirements emerge
- State extraction formula: For device at index i, state equals (bitmask right-shift (i times 2)) bitwise-AND 0b11
- State setting formula: Clear 2 bits at position (i times 2) using bitwise-AND with complement of (0b11 left-shift (i times 2)), then bitwise-OR with (new_state left-shift (i times 2))
- Transition cost calculation: sum over all devices of transition_cost_matrix[old_state][new_state] for that device
