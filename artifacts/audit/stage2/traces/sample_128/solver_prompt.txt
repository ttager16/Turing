### Problem Statement

In a large-scale urban planning project, a city planner now faces a more intricate challenge: rather than simply identifying a single "center" in each district's tree, they must juggle multiple interacting tree-based districts that evolve under different scenarios. Each district is still represented by a tree graph, but with additional layers of complexity:

1. Intermittent node deactivations that temporarily remove certain nodes (and their incident edges) from service.
2. Dynamic edge weight updates reflecting changing road lengths due to construction or closures.
3. Queries that merge two or more tree districts into a shared service region if they become interconnected at specific nodes.

In this advanced setting, the goal is to maintain an optimal "emergency response hub" for each district (or newly merged region) under these offline modifications. The objective is no longer a simple minimization of maximum distance, each node carries both a traffic density factor and a capacity constraint indicating how many emergencies it can handle at a time. Placing a service hub at a node must ensure minimal overall response time while respecting each node's capacity limits and the shifting topology of the network.

The planner must design a robust offline algorithm in Python that processes these tree-based regions, handles all queued modifications (node deactivations/activations, edge updates, district merges), and, after all modifications are applied, computes the final best hub placement(s). Due to the complex interplay of constraints and multi-step updates, straightforward brute-force or naive tree-center formulas are inadequate. Advanced data structures (like Heavy-Light Decomposition or DS-trees) and tree-based dynamic programming may be required to maintain and update relevant metrics (distance, capacity usage, traffic distribution) for each scenario efficiently.

### Objectives

1. Develop a Python function that, given multiple tree districts alongside a sequence of offline events, returns the final optimal hub node in each modified tree/district.
2. Incorporate an efficient mechanism to manage node/edge updates and inter-district merges, ensuring that recalculations do not degrade for large or frequently updated trees.
3. Track node capacity usage in tandem with road lengths (edge weights) and traffic densities (node weights) to compute a **compound objective function** reflecting realistic emergency response needs.
4. Demonstrate scalability to handle up to tens of thousands of nodes across multiple districts, including corner cases where merges form large, interconnected super-districts.

## Compound Objective Function

To determine the **optimal hub** node `h` within a district or merged region `D = (V, E)`, minimize

[
F(h) = \sum_{i \in V_{active}} \frac{T_i \times d(h, i)}{C_h}
]

Where:

* (T_i) is the traffic density (node weight) at node (i).
* (d(h, i)) is the shortest path distance (sum of edge weights) between hub (h) and node (i).
* (C_h) is the capacity of hub node (h).
* (V_{active}) is the set of active nodes in the district after applying deactivations.

The optimal hub is the node `h` that minimizes (F(h)). If multiple nodes yield the same minimum value, choose the one with the smallest node ID.

This function balances travel time and hub efficiency: higher-capacity hubs distribute load more effectively, while nodes with higher traffic density contribute more heavily to total response time.

## Input

The function takes a list of district dictionaries.
Each dictionary describes one district, which is a separate tree with its own roads, nodes, and events.

Every district dictionary must have these keys:

* `"nodes"`: an array of numbers listing all node IDs in that district.
* `"edges"`: an array of connections between nodes. Each connection is `[u, v, weight]`, where `u` and `v` are node IDs and `weight` is the distance or cost between them.
* `"node_weights"`: an object that maps **stringified node IDs** to integer traffic densities. Keys must be strings.

  * Example: `"node_weights": { "1": 10, "2": 20, "3": 30 }`
* `"capacity"`: an object that maps **stringified node IDs** to integer capacities. Keys must be strings.

  * Example: `"capacity": { "1": 2, "2": 2, "3": 3 }`
* `"events"`: an array of actions that happen in the district. Each event is an array describing the operation.

**Important on IDs:** Everywhere else in the input where node IDs appear inside arrays or scalars (for example in `"nodes"`, `"edges"`, and `"events"`), they are numbers. Only the keys of `"node_weights"` and `"capacity"` are strings.

**Event types:**

* `["deactivate_node", node_id]` means temporarily remove a node and its edges.
* `["activate_node", node_id]` means restore a previously deactivated node.
* `["update_road", node_u, node_v, new_weight]` updates the weight of an existing edge.
* `["merge", district_index_to_merge, merge_node_in_this_district, merge_node_in_other_district, connecting_edge_weight]` merges this district with another, creating a new edge of the specified weight between the given nodes.

After all events are processed, each district (or merged super-district) is evaluated to determine its optimal hub node according to the compound objective function.

## Output

Return a list of integers with the same length as the input. Each position corresponds to a district in the input:

* `node_id`: ID of the final optimal hub for that district or merged region if it is the surviving district.
* `-1`: the district was merged into another and no longer exists independently.

**Example interpretation:** If two districts merge, the surviving district's entry holds the computed hub ID, while the merged-into district's entry is `-1`.
