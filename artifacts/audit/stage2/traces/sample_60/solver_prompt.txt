### Problem Statement

In the context of a real-time stock trading platform, the system must manage a **partially persistent, concurrency-aware data structure** that integrates adaptive balancing with interval-based indexing. This structure should efficiently handle rapid insertions, deletions, updates, and compound range queries across historical stock price data, while maintaining internal consistency under simulated concurrent operations and partial rollbacks.

Rather than relying on true multithreading, **concurrency should be simulated logically** — for example, by tracking the order of operations and supporting rollback mechanisms to revert previous updates. The data structure should dynamically adjust its organization to keep operations efficient, even as overlapping updates or queries occur.

---

### Objectives

1. **Adaptive Multi-Layer Balancing**
   Maintain optimal height by dynamically distributing nodes across multiple tiers (e.g., segment tree overlay or interval-based indexing). The structure should self-balance after single-event or bulk operations to ensure minimal query and update latency.

2. **Compound Range Query Support**
   Implement advanced range queries that return aggregated metrics — such as the maximum or minimum stock price less than or equal to a given value — while efficiently updating internal metadata.

3. **Simulated Concurrent Transactions and Partial Rollbacks**
   Handle logically concurrent operations by maintaining a transaction history that supports partial rollbacks. This ensures data consistency when recent updates must be undone.

4. **Stateful Operation Tracking**
   Store metadata at internal nodes (e.g., last update timestamp, trade volume, or risk coefficient). These fields should be updated or queried within `O(log n)` time, similar to normal stock operations.

5. **Scalability Under High Frequency**
   The data structure must scale to handle a large number of operations while keeping memory and runtime overhead proportional to the logarithm of the total node count.

---

### Constraints

* The algorithmic complexity of all operations (insert, delete, update, query, rollback) must be **O(log n)**.
* The design must handle edge cases such as negative or zero stock prices, repeated rollbacks, and overlapping intervals.
* Only **built-in Python functionality** may be used; no external libraries or frameworks are allowed.
* Concurrency is **simulated**, not physically parallel (no threading or multiprocessing).
