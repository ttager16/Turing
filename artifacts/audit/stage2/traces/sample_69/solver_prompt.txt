Problem Statement:
## Urban Traffic Signal Optimization Using Dynamic Graph Buffering

An urban transportation management system dynamically adjusts traffic lights based on periodically updated traffic information.
The system models the city’s road network as a directed graph, where intersections and roads are represented as nodes and edges.
Through dynamic graph buffering, the system continuously integrates new data on traffic density, lane blockages, and emergency activity to improve signal timing and reduce congestion.


### Scenario

The city’s traffic network is a directed graph in which:
* Each intersection represents a traffic signal.
* Each road represents a one-way connection to another intersection.
* Road capacities exist in the real system but are not used in this specific computation.
Periodic updates report per-intersection information on traffic "density", "accidents", and "priority requests" (for example, emergency vehicles).
Using this information, the system must determine two outputs for each intersection:
* The optimized green-signal duration (in seconds).
* An overflow indicator showing whether congestion is likely.


### Objectives

1. **Graph Representation and Data Updates**  
   - The city network is modeled as a **directed graph** where each intersection lists its outgoing connections.  
   - Each traffic record corresponds to one intersection and may include:  
     * traffic_density (float 0.0–1.0), accidents (e.g., ["LANE_OUTAGE"]), and priority_requests (e.g., ["EMERGENCY"]).  
   - If multiple records exist for the same intersection, compute each record’s preliminary timing and keep the one with the lowest value.  
   - If tied, prefer a record containing an EMERGENCY request, if still tied, use the last record.  
   - Intersections mentioned in traffic_data but not present in the graph are added with empty outgoing lists.  
   - Missing fields default as follows:  
     * traffic_density -> no traffic data (neighbor-average rule), accidents or priority_requests -> empty list.

2. **Preliminary Signal Duration**
   For intersections with traffic data:
   - Start from a base duration of 20 s.  
   - Add "70 × traffic_density", rounded half-up.  
   - Add 10 s if an EMERGENCY request is present.  
   - If traffic_density > 0.7 and "LANE_OUTAGE" is reported, halve the total (after the emergency bonus) using floor division.  
   For intersections without traffic data, use the half-up average of already-computed outgoing neighbors. if none are known, or no outgoing edges exist, use 45 s.

3. **Normalization**
   - After all preliminary durations are known, take the union of graph keys and traffic-data intersections.  
   - Scale all durations so their mean (before rounding) equals 45 s.  
   - Then round half-up and clamp each value to 20–90 s.  
   - The final mean after rounding may differ slightly from 45 s.

4. **Overflow Determination**  
   - An intersection is marked overflowing if its optimized duration is below 30 s, or if its traffic_density is > 0.75 without an EMERGENCY request.  
   - Intersections lacking traffic data consider only the first condition.
