Problem Statement:
### Problem Statement
We manage a region-wide traffic network. Intersections are nodes with internal fields: lane counts, sensor confidence, priority class, and emergency flags. Roads are directed edges with capacity and travel time. The goal is to maintain a dynamic partial topological order of intersections that reflects real-time edge and node state changes, supports multi-phase signal coordination, and allows emergency overrides.

The system must ingest streaming updates that change edge existence, edge capacity, and node state. The maintained order must remain valid for downstream modules that perform congestion forecasting and signal scheduling while updates arrive at high rate.

---

### Objectives
1. Dynamic topological ordering algorithm requirements:
  - Support node state fields: "priority" (int), "emergency" (bool), "phases" (list of ints).
  - Support weighted directed edges where weight includes capacity and travel time.
  - Apply updates incrementally: add/remove edges, change capacity, or change node state without full recomputation when possible.
  - Detect cycles created by a batch and abort the batch by raising a RuntimeError whose message begins with
    `CycleDetectedError` and lists the nodes in the cycle.
    - priority is a soft tie-breaker and must not override explicit directed edges.

2. Ingestion and reconciliation:
  - Updates arrive as timestamped records and should be applied in batch order.
  - Conflicting updates are resolved according to a `reconcile_policy` parameter. Default policy is "abort_on_conflict" which cancels the batch and returns an error. An alternate policy is "ignore_with_skip" which skips invalid operations silently.
    - Batches are atomic: under "abort_on_conflict" either the entire batch applies or none of it applies.

3. Consistency guarantees:
  - If `emergency` implies precedence, the prompt must specify the exact precedence edges to add; otherwise require explicit precedence updates for emergency handling.
  - When updates are asynchronous, the algorithm must either produce a consistent ordering for the latest applied batch or return a clear error describing the inconsistency.

---

### Constraints
- The network may contain up to 10 nodes and 10 directed edges.
- Implementations must use only the Python standard library.
- The entire implementation must live inside a single top-level function `update_topological_order` (helper functions or classes may be nested inside it). Only module-level imports are allowed outside the function.
- Performance: target O(V + E) for full recomputation where V is nodes and E is edges; incremental updates should be faster when fewer than half the nodes are affected.
- Determinism: when multiple valid topological orders exist, the function must use node `priority` (lower value first) as a tie-breaker, then node id (ascending) to ensure reproducible results.

### Inputs
#### Validation rules
The function must validate inputs before applying updates. On invalid input the function should return an error string with the exact format below. All error messages must match the formats exactly.

- Graph validation:
  - If `graph` is not a dict mapping string node ids to dicts of string->int, return string:
    "TypeError: InvalidInputError: 'graph' must be Dict[str, Dict[str, int]]"
  - If the graph has more than 10 nodes or more than 10 total edges, return string:
    "ValueError: InputSizeError: graph has {n} nodes and {m} edges; limits are max_nodes=10, max_edges=10"

- Node states validation:
  - If `node_states` is not a dict mapping string node ids to dict, return string:
    "TypeError: InvalidInputError: 'node_states' must be Dict[str, Dict[str, Any]]"
  - If any node in `graph` is missing from `node_states`, return string:
    "KeyError: InvalidInputError: missing node_states entry for node {node_id}"

- Updates validation:
  - If `updates` is not a list, return string:
    "TypeError: InvalidInputError: 'updates' must be a list of update records"
  - Each update must be a dict. If an update at index {i} is not a dict, return string:
    "TypeError: InvalidUpdateError: update at index {i} must be a dict"
  - Required keys and types per update record:
    - "op": str; allowed values: "add_edge", "remove_edge", "capacity_change", "change_state". If invalid, return string:
      "ValueError: InvalidUpdateError: update at index {i} has invalid op '{op}'"
    - "src": str for edge and state ops. If missing or wrong type, return string:
      "TypeError: InvalidUpdateError: update at index {i} missing or invalid 'src'"
    - "dst": str for edge ops (add/remove/capacity_change). If missing or wrong type, return string:
      "TypeError: InvalidUpdateError: update at index {i} missing or invalid 'dst'"
    - "value": int for capacity_change; dict for change_state. If type mismatch, return string:
      "TypeError: InvalidUpdateError: update at index {i} has invalid 'value' for op '{op}'"
    - "ts": optional; if present must be int or float. If not, return string:
      "TypeError: InvalidUpdateError: update at index {i} has invalid 'ts' type; must be int or float"

- Semantic update errors (when reconcile_policy is "abort_on_conflict", return error string; when "ignore_with_skip", skip silently):
  - Reference to unknown node: "RuntimeError: InvalidUpdateError: update at index {i} references unknown node {node_id}"
  - Edge does not exist for capacity_change or remove_edge: "RuntimeError: InvalidUpdateError: update at index {i} references non-existent edge {src}->{dst}"
  - change_state value not a dict: "TypeError: InvalidUpdateError: update at index {i} change_state 'value' must be a dict"

- Reconcile policy validation:
  - If `reconcile_policy` is not one of the allowed strings, return string:
    "ValueError: InvalidInputError: reconcile_policy must be 'abort_on_conflict' or 'ignore_with_skip'"

- Cycle detection:
  - If applying the batch would introduce a cycle and `reconcile_policy` is "abort_on_conflict", return string:
    "RuntimeError: CycleDetectedError: batch creates cycle involving nodes {nodes}; failed_update_index={i}"

  '4': {"priority": 1, "emergency": False},
  '5': {"priority": 1, "emergency": False},
  '6': {"priority": 1, "emergency": False}

**Expected output:** `['1', '3', '2', '4', '5', '6']` (deterministic ordering after applying updates)

**Expected behavior:** Invalid update is skipped silently; ordering remains unchanged.
