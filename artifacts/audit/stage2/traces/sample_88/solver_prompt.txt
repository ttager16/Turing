Problem Statement:
### Problem Statement
A high-frequency trading firm needs an advanced data structure that combines segment trees with dynamic connectivity to handle real-time market data. The system must support:
1. **Point updates** with cascade propagation through dependent indices
2. **Range queries** with multiple aggregation modes (sum, max, min, average)
3. **Dynamic grouping** where stocks can be linked/unlinked, affecting query results
4. **Historical versioning** where each update creates a checkpoint that can be queried
5. **Conditional range updates** that modify multiple elements based on criteria
The key challenge: updates to one stock can trigger cascading updates to all stocks in its dependency chain, and queries must account for these transitive relationships while maintaining O(log n) performance.

### Objectives
1. **Implement a Versioned Segment Tree with Persistent Structure:**
   - Support point updates that create new versions
   - Allow queries on any historical version (0 to current)
   - Each version must be independently queryable without affecting others
2. **Integrate Dynamic Connectivity (Union-Find with Versioning):**
   - Track which stocks are in the same group
   - Support linking two stocks (union operation)
   - Support unlinking a stock from its group (make it singleton)
   - Group membership affects range query results
3. **Implement Cascading Updates:**
   - When updating a stock, all stocks in its group receive proportional updates
   - The proportion is based on their initial values relative to the updated stock
   - Cascading must be atomic and create only one new version
4. **Support Multiple Aggregation Modes:**
   - Range queries can compute: sum, max, min, or average
   - Mode is specified per query
   - Results must be exact integers (use integer division for average)
5. **Handle Conditional Range Updates:**
   - Update all elements in range [i, j] where element value satisfies a condition
   - Conditions: greater_than, less_than, equal_to, between
   - Each conditional update creates one new version
   
### Constraints
- 1 ≤ n ≤ 10,000 (number of stocks)
- 1 ≤ q ≤ 10,000 (number of queries)
- 0 ≤ stock values ≤ 1,000,000
- All values are non-negative integers
- Version numbers start at 0 (initial state)
- Must support at least 1,000 versions
- Time complexity: O(log n) for point queries, O(log n + k) for range queries of size k
- Space complexity: O(n × versions) is acceptable
- Must be thread-safe for concurrent read operations (writes are sequential)

### Critical Implementation Requirements

1. **Cascading Updates Must Be Proportional:**
   - If updating stock A from value X to Y in a group {A, B, C}
   - Ratio = Y / X
   - All other stocks in group multiply their values by this ratio
   - Use integer arithmetic: new_value = (old_value × Y) // X
   
2. **Version Independence:**
   - Querying version V must return exact state after V-th update
   - Later versions must not affect earlier version queries
   - Use persistent data structure or copy-on-write
   
3. **Group Dynamics:**
   - Initial state: each stock is in its own singleton group
   - Link operation: merge groups using union-find
   - Unlink operation: remove one stock, others stay together
   - Groups persist across versions correctly
   
4. **Aggregation Modes:**
   - sum: simple addition
   - max: maximum value in range
   - min: minimum value in range
   - avg: integer division (sum // count)
   
5. **Conditional Updates:**
   - Check each element in range individually
   - Update only those satisfying condition
   - All updates in single conditional are atomic (one version)
   
6. **Edge Cases:**
   - Empty ranges (start > end): return 0
   - Invalid version: return 0
   - Invalid index: skip operation
   - Division by zero in cascade: skip cascade
   - Self-linking: no-op
   - Unlinking already singleton: no-op
